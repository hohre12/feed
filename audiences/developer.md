# Developer

## Search Sources

### No Topic (Content Discovery)
1. GitHub Trending (today/this week trending repositories)
2. Hacker News Top/Best
3. Dev.to Trending
4. Reddit r/programming, r/webdev

### With Topic (Fact Verification)
1. GitHub (repositories, star count, latest releases)
2. Official documentation (docs for the relevant technology)
3. Stack Overflow
4. Tech blogs (official blogs, engineering blogs)

## Tone
- Use technical terms as-is (CI/CD, ORM, bundler — no paraphrasing)
- "I tried this and..." hands-on practitioner tone
- No excessive hype ("Revolutionary AI tool!" is wrong — "This is actually more useful than I expected" is right)
- Developer memes/humor are welcome ("overtime", "legacy code", "it works on my machine")

## Terminology Level
- Technical jargon used without explanation is fine
- Abbreviations are fine (PR, CI, DB, API, etc.)
- Short code snippets can be included

## Recommended Styles
- **insight**: Tech tips, new tool introductions, GitHub trending picks
- **curated-list**: Developer tool lineups, library recommendations
- **relatable**: Developer daily life (bugs, overtime, code reviews)
- **self-deprecation**: Developer self-roasts (legacy code, tech debt)

## Not Recommended Styles
- **twist-poem**: The emotional monologue structure feels out of place for technical content. Personifying a database or a CI pipeline rarely lands with developer audiences who prefer directness.
- **quote-parody**: Classic proverbs and famous quotes have weak overlap with developer culture. The humor gap between the original quote and the punchline is harder to bridge with technical subject matter.
